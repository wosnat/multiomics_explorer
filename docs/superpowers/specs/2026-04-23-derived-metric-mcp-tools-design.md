# DerivedMetric MCP surface — Slice 1: new tools

**Status:** design, approved to write plan
**Date:** 2026-04-23
**Scope:** slice 1 of 3 (new MCP tools only; existing-tool rollups and `compartment` retrofit deferred)

## Context

The knowledge graph now represents non-DE evidence as `DerivedMetric` nodes with three measurement edge types dispatched by `value_kind` (numeric / boolean / categorical), parallel to the existing `ClusteringAnalysis` pattern. Full KG details:
`multiomics_biocypher_kg/docs/kg-changes/non-de-evidence-extension.md`.

Per-DerivedMetric/Experiment/Publication/Organism/Gene rollups are already materialized by the post-import step; all new int props default to `0` and `str[]` props to `[]` (never null).

Baselines as of 2026-04-23:
- 13 DerivedMetric nodes (7 Biller 2018 + 6 Waldbauer 2012)
- `Derived_metric_flags_gene` edges: 4,160 (Biller 2018, 6 boolean DMs)
- `Derived_metric_classifies_gene` edges: 258 (Biller 2018, 1 categorical DM)
- `Derived_metric_quantifies_gene` edges: 1,872 (Waldbauer 2012, 6 numeric × 312 genes)

## The three-slice split

| Slice | Scope | Status |
|---|---|---|
| **1** | 5 new MCP tools built from scratch, using already-materialized post-import props | **this spec** |
| 2 | Extend existing tools (`gene_overview`, `list_experiments`, `list_publications`, `list_organisms`, `list_filter_values`) with DM rollup fields — use clustering surface as template | deferred |
| 3 | `Experiment.compartment` retrofit — mirror the `background_factors` pattern wherever it appears (filter / summary / field) across `list_*` and DE tools | deferred |

Slice 1 is self-contained: it adds new tools without touching existing ones.

## Scope — what slice 1 covers

Three primary research questions:

- **Discovery** — what non-DE evidence exists for this organism / publication / experiment?
- **Gene-centric lookup** — given a locus_tag list, what derived-metric annotations does each gene carry?
- **Ranking / drill-down** — given a metric, which genes are flagged / top-ranked / in a category?

Explicitly **out of slice 1**: cross-evidence integration (combining DM + DE + clusters in one view), enrichment-ready gene-set hooks (TERM2GENE from DM buckets). These may ship later, possibly outside MCP.

## Architecture

Three-layer, mirroring the clustering pattern:

| Layer | File | Contribution |
|---|---|---|
| Query builders | `multiomics_explorer/kg/queries_lib.py` | 10 new builders (1 summary + 1 detail per tool × 5 tools) |
| API functions | `multiomics_explorer/api/functions.py` | 5 new public functions; 2-query pattern (summary always runs, detail skipped when `limit=0`); envelope helpers shared with clustering tools |
| MCP wrappers | `multiomics_explorer/mcp_server/tools.py` | 5 new `@mcp.tool` wrappers; Pydantic `{Name}Result` / `{Name}Response` models |
| About content inputs | `multiomics_explorer/inputs/tools/{name}.yaml` | 5 new YAML authoring files |
| About content outputs (generated) | `multiomics_explorer/skills/multiomics-kg-guide/references/tools/{name}.md` | generated by `scripts/build_about_content.py` |
| Analysis-layer reference | `multiomics_explorer/skills/multiomics-kg-guide/references/analysis/derived_metrics.md` | new concept + cross-tool workflow guide |

Tool-to-clustering-analog map:

| New tool | Clustering analog | Purpose |
|---|---|---|
| `list_derived_metrics` | `list_clustering_analyses` | Discovery |
| `gene_derived_metrics` | `gene_clusters_by_gene` | Gene-centric batch |
| `genes_by_numeric_metric` | `genes_in_cluster` | Drill-down, numeric |
| `genes_by_boolean_metric` | `genes_in_cluster` | Drill-down, boolean |
| `genes_by_categorical_metric` | `genes_in_cluster` | Drill-down, categorical |

### Why three drill-down tools instead of one

The filter surface and return columns are genuinely different per `value_kind`:

- numeric → `min/max_value`, `min/max_percentile`, `bucket`, `significant_only`, `max_adjusted_p_value`, `max_rank`; returns `value, rank_by_metric, metric_percentile, metric_bucket, adjusted_p_value, significant`
- boolean → `flag`; returns `value` (`"true"|"false"`)
- categorical → `categories`; returns `value` (category string)

A single polymorphic tool would require ~9 usually-null params plus per-call dispatch logic and would make the MCP description harder for the LLM to reason about. Three focused tools each have a homogeneous return and a clean filter surface. Total tool-slot cost: 5.

### Selection semantics

`metric_type` is a *category tag*, not a primary key — it can repeat across publications. Every tool that selects DMs accepts either:

- `derived_metric_ids: list[str]` — unambiguous, by node ID (from `list_derived_metrics`)
- `metric_types: list[str]` — category-style; unions every DM with that tag, further narrowed by scoping filters (organism, compartment, publication, experiment)

For drill-downs, these two params are mutually exclusive; exactly one required. For discovery/gene-centric tools, `derived_metric_ids` and `metric_types` are both optional filters and may be combined with each other and other filters.

`name` stays out of the filter surface (not guaranteed unique) — it appears in results for humans.

## Tool 1 — `list_derived_metrics`

Discovery. Mirrors `list_clustering_analyses` shape.

### Parameters

```
search_text: str | None                  # Lucene on derivedMetricFullText (name, field_description)
organism: str | None
metric_types: list[str] | None
value_kind: str | None                   # 'numeric' | 'boolean' | 'categorical'
compartment: str | None                  # 'whole_cell' | 'vesicle' | 'exoproteome' | 'spent_medium' | 'lysate'
omics_type: str | None                   # includes 'PAIRED_RNASEQ_PROTEOME'
treatment_type: list[str] | None
background_factors: list[str] | None
growth_phases: list[str] | None
publication_doi: list[str] | None
experiment_ids: list[str] | None
derived_metric_ids: list[str] | None
rankable: bool | None
has_p_value: bool | None
summary: bool = False                    # sugar for limit=0
verbose: bool = False
limit: int | None
offset: int = 0
```

### Envelope

```
{
  total_entries, total_matching,
  by_organism, by_value_kind, by_metric_type, by_compartment,
  by_omics_type, by_treatment_type, by_background_factors, by_growth_phase,
  score_max, score_median,           # only when search_text provided
  returned, offset, truncated,
  results: [...]
}
```

### Result columns

**Compact:** `derived_metric_id, name, metric_type, value_kind, rankable, has_p_value, unit, allowed_categories, field_description, organism_name, experiment_id, publication_doi, compartment, omics_type, treatment_type, background_factors, total_gene_count, growth_phases` plus `score` when `search_text` is provided.

**Verbose adds:** `p_value_threshold, treatment, light_condition, experimental_context`.

Rationale for compact inclusions: `unit`, `allowed_categories`, and `field_description` are essential for interpreting a DM in the first place — `metric_type` codes are often opaque (e.g. `diel_amplitude_protein_log2`), so the discovery payload has to carry enough context for an LLM to triage without a round-trip to verbose.

## Tool 2 — `gene_derived_metrics`

Gene-centric batch. Mirrors `gene_clusters_by_gene` — single organism enforced, with `not_found` / `not_matched` reporting.

### Parameters

```
locus_tags: list[str]                    # required; single-organism enforced
organism: str | None
metric_types: list[str] | None
value_kind: str | None
compartment: str | None
treatment_type: list[str] | None
background_factors: list[str] | None
publication_doi: list[str] | None
derived_metric_ids: list[str] | None
summary: bool = False
verbose: bool = False
limit: int | None
offset: int = 0
```

### Envelope

```
{
  total_matching,                        # rows (gene × DM pairs)
  total_derived_metrics,                 # distinct DMs touching these genes
  genes_with_metrics, genes_without_metrics,
  not_found, not_matched,                # locus_tag echo
  by_value_kind, by_metric_type,
  by_compartment, by_treatment_type, by_background_factors,
  by_publication,
  returned, offset, truncated,
  results: [...]
}
```

### Result columns (one row per gene × DM; sparse across value_kinds)

**Compact (always):** `locus_tag, gene_name, derived_metric_id, metric_type, value_kind, name, field_description, unit, allowed_categories, value, compartment, treatment_type, background_factors, publication_doi`

- `value` is polymorphic:
  - numeric rows → `float`
  - boolean rows → `"true"` | `"false"`
  - categorical rows → category string
- Consumers branch on `value_kind`.

**Compact (numeric-only; null otherwise):** `metric_percentile, metric_bucket, adjusted_p_value, significant, rank_by_metric`

These carry real signal for ranking/significance with no analog in other kinds — keeping them as sparse columns avoids both a nested dict (awkward for flat tables) and lossy collapse.

**Verbose adds:** `treatment, light_condition, experimental_context, p_value` (raw).

### Not-found / not-matched

- `not_found` — `locus_tags` absent from the KG
- `not_matched` — `locus_tags` present in the KG but with zero DM annotations passing the filters

## Tool 3 — `genes_by_numeric_metric`

Drill-down on `Derived_metric_quantifies_gene`. Mirrors `genes_in_cluster`.

### Parameters

```
# Selection — exactly one required
derived_metric_ids: list[str] | None
metric_types: list[str] | None

# Scoping (intersected with selected DM set)
organism: str | None
locus_tags: list[str] | None             # restrict to gene subset
experiment_ids: list[str] | None
publication_doi: list[str] | None
compartment: str | None
treatment_type: list[str] | None
background_factors: list[str] | None
growth_phases: list[str] | None

# Kind-specific edge filters
min_value: float | None
max_value: float | None
min_percentile: float | None
max_percentile: float | None
bucket: list[str] | None                 # subset of {top_decile, top_quartile, mid, low}
significant_only: bool = False
max_adjusted_p_value: float | None
max_rank: int | None                     # implicitly requires rankable=true DMs

summary: bool = False
verbose: bool = False
limit: int | None
offset: int = 0
```

### Validation

- All selected DMs must have `value_kind='numeric'`; otherwise `ValueError` listing offending IDs + their actual `value_kind`.
- `max_rank` requires all selected DMs have `rankable=true`; otherwise `ValueError` listing offenders.

### Envelope (shared across all 3 drill-down tools)

```
{
  total_matching,                        # rows (gene × DM)
  total_derived_metrics,                 # DMs after selection + scoping
  by_organism, by_metric_type,
  by_compartment, by_publication, by_experiment,
  top_categories,                        # gene_category rollup
  genes_per_metric_max, genes_per_metric_median,
  not_found_ids, not_matched_ids,        # from derived_metric_ids
  not_found_metric_types, not_matched_metric_types,
  not_matched_organism,
  returned, offset, truncated,
  results: [...]
}
```

### Result columns

**Shared compact (all 3 drill-downs):** `locus_tag, gene_name, product, gene_category, organism_name, derived_metric_id, metric_type, name, compartment, experiment_id, publication_doi, treatment_type, background_factors`

**Numeric extras (compact):** `value, rank_by_metric, metric_percentile, metric_bucket, adjusted_p_value, significant`

**Verbose adds:** `gene_function_description, gene_summary, field_description, unit, treatment, light_condition, experimental_context, p_value`.

### Default sort

`r.rank_by_metric ASC NULLS LAST, r.value DESC` — rank-ordered when available, falling back to raw value for non-rankable DMs.

## Tool 4 — `genes_by_boolean_metric`

Drill-down on `Derived_metric_flags_gene`.

### Parameters

Same selection + scoping block as Tool 3. Kind-specific:

```
flag: bool | None                        # maps to r.value_flag = 'true' | 'false'
                                         # default None returns both; typical caller passes True
```

### Validation

All selected DMs must have `value_kind='boolean'`.

### Result columns

Shared compact (see Tool 3). Extras:

- **Compact:** `value` — `"true"` or `"false"` (string, matches edge storage)
- **Verbose:** no extras (boolean edges have no additional props)

### Default sort

`locus_tag ASC` — no score to rank by.

## Tool 5 — `genes_by_categorical_metric`

Drill-down on `Derived_metric_classifies_gene`.

### Parameters

Same selection + scoping block as Tool 3. Kind-specific:

```
categories: list[str] | None             # filter on r.value_text IN [...]
                                         # must be subset of union of parent allowed_categories
```

### Validation

- All selected DMs must have `value_kind='categorical'`.
- `categories` must be subset of the union of parent `allowed_categories`; unknown values → `ValueError` listing offenders plus the allowed set for context.

### Result columns

Shared compact (see Tool 3). Extras:

- **Compact:** `value` — the category string
- **Verbose:** no extras

### Default sort

`r.value_text ASC, locus_tag ASC` — stable grouping by category.

## Cross-cutting validation

| Situation | Behavior |
|---|---|
| Empty `locus_tags` (required param) | `ValueError` (matches existing batch tools) |
| `summary=True` | Force `limit=0`; detail query skipped |
| `search_text=""` | `ValueError` |
| Lucene parse error in `search_text` | Retry with special chars escaped via `_LUCENE_SPECIAL` (matches existing pattern) |
| Both `derived_metric_ids` AND `metric_types` on drill-downs | `ValueError` "provide one, not both" |
| Neither selection param on drill-downs | `ValueError` "must provide one of …" |
| Selected DM has wrong `value_kind` for the tool | `ValueError` listing offending `derived_metric_id`s + their actual `value_kind` |
| `max_rank` used but some selected DMs have `rankable=false` | `ValueError` listing offenders |
| `categories` includes value not in any parent `allowed_categories` | `ValueError` listing unknowns + allowed-set context |
| `value_kind` filter receives value not in `{numeric, boolean, categorical}` | `ValueError` |
| Single-organism check fails (locus_tags span organisms, or explicit `organism` conflicts with inferred) | `ValueError` via `_validate_organism_inputs` |
