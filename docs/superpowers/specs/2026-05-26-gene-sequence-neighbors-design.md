# Gene sequence + genomic-neighbor MCP tools

**Status:** design, approved to write plan
**Date:** 2026-05-26
**Scope:** 2 new MCP tools (`gene_aa_sequence`, `gene_neighbors`). No KG-side changes — both read existing `Gene` node properties. No new query patterns beyond contig-scoped ordering.

## Context

Both tools read properties already present on every `Gene` node (verified live 2026-05-26 against the freshly-rebuilt KG):

- `sequence: str` — **amino-acid** sequence (starts with `M`; length ≈ `(end − start)/3`). There is no separate nucleotide sequence stored.
- `contig: str`, `start: int`, `end: int` — always populated **together**.
- `strand: str` — `'+'` / `'-'`, **populated for only ~50% of genes** (nullable, sparse).
- `protein_id`, `gene_name`, `product`, `gene_category`, `organism_name` — identity / functional context.

Coverage and ordering facts that shape the design:

- **`sequence`/`start`/`contig` co-occur but are not universal** — ~10–15% of genes lack them (e.g. *Prochlorococcus* MIT9303: 2,587 / 3,114). These are expression-only genes without a genome-annotation match → both tools need a `not_matched` bucket distinct from `not_found`.
- **locus_tags are globally unique** (0 ambiguous across all organisms) → both tools accept cross-organism batch input; no single-organism enforcement needed.
- **`(contig, start)` is unique** within every contig → ordering by `start` is deterministic, so rank offsets compute cleanly with no tie-breaking.
- Genomes are often fragmented (e.g. *Alteromonas (MarRef v6)*: 573 contigs) → neighbor windows **must** scope to the same `contig` **and** `organism_name`.

The KG now spans many more organisms than Prochlorococcus/Alteromonas (Pseudomonas, Ruegeria, Synechococcus, Marinobacter, Shewanella, …). Neither tool assumes a particular organism set.

## Driving use cases

- **`gene_aa_sequence`** — export amino-acid sequences for external tools (BLAST / HMMER / alignment). Output is copy-paste-oriented (FASTA), not in-graph reasoning.
- **`gene_neighbors`** — inspect a gene's genomic neighborhood for operon / synteny reasoning inside the KG: what's adjacent, on which strand, with what function.

## Architecture

Two gene-centric batch lookups, peers of `gene_details` / `gene_overview` / `gene_homologs`. Standard 4-layer build.

### Tool inventory

| Tool | Direction | Pattern peer | Status |
|---|---|---|---|
| `gene_aa_sequence` | gene → AA sequence (batch, cross-organism) | `gene_details` | new |
| `gene_neighbors` | gene → genomic neighbors (batch, cross-organism, flat long) | `gene_homologs` | new |

### Layer table

| Layer | File | Contribution |

No KG asks. No new indexes (lookups are by `locus_tag`, already indexed; neighbor scan is by `contig` + `start` on a small per-organism set).

## Tool 1 — `gene_aa_sequence`

Export-optimized amino-acid sequence lookup. Single-node read of `Gene` properties only — no traversals.

### Parameters

| Param | Type | Default | Notes |
|---|---|---|---|
| `locus_tags` | `list[str]` | — (required) | Batch input. Cross-organism OK (locus_tags globally unique). |
| `fasta` | `bool` | `False` | Selects payload mode (below). |
| `limit` | `int` | `25` | Sequences are the heaviest payload in the surface — paginate by default. |
| `offset` | `int` | `0` | Pagination. |

### Payload modes — sequence never appears twice

- **`fasta=False`** → `results` rows each carry the raw sequence:
  `locus_tag`, `organism_name`, `gene_name`, `product`, `protein_id`, `sequence_length`, `sequence`.
- **`fasta=True`** → `results` rows carry the **same metadata with `sequence` set to `null`** (key present for stable row shape; `sequence_length` still populated), **plus** an envelope `fasta` field: one multi-FASTA blob covering the returned page.
  - FASTA record header: `>{locus_tag} {organism_name}|{protein_id}|{product}`.
  - Rationale: returning per-row `sequence` *and* a concatenated blob ships every sequence twice — wasteful on the heaviest field. The flag picks one carrier.

### Envelope

- `total_matching` — count of input locus_tags that resolved to a gene **with** a sequence.
- `truncated` — bool, whether `limit` cut the result set.
- `by_organism` — `{organism_name: count}` rollup.
- `sequence_length_stats` — `{count, min, q1, median, q3, max, mean}` over **all matched genes with a sequence** (the full matched set, independent of `limit`/`offset`, so it's stable across pages). Mirrors the `value_min/q1/median/q3/max` convention on `DerivedMetric` / `MetaboliteAssay`. Present in both `fasta` modes.
- `not_found` — `list[str]` of input locus_tags absent from the KG.
- `not_matched` — `list[str]` of locus_tags that resolve to a gene but whose `sequence` is null (the ~10–15%).

### Edge cases

- Gene exists but `sequence` is null → `not_matched`, never an empty/placeholder row.
- `fasta=True` with all inputs `not_matched`/`not_found` → empty `results`, `fasta` is an empty string, `sequence_length_stats.count = 0`.

## Tool 2 — `gene_neighbors`

Genomic-neighborhood lookup for operon / synteny context. **Flat long format** — one row per (anchor × neighbor), mirroring `gene_homologs`.

### Parameters

| Param | Type | Default | Notes |
|---|---|---|---|
| `locus_tags` | `list[str]` | — (required) | Batch anchors. Cross-organism OK. |
| `window` | `int` | `5` | ±N genes by rank on the same contig (N upstream + N downstream). |
| `max_bp_distance` | `int \| None` | `None` | Optional cap: drop neighbors whose intergenic gap to the anchor exceeds this many bp. Trims neighbors across assembly gaps. |
| `same_strand` | `bool \| None` | `None` | `None` = all neighbors; `True` = co-oriented only; `False` = opposite-strand only. See strand handling. |

### Neighbor computation

For each anchor: scope to genes on the same `contig` **and** `organism_name` with non-null `start`, order by `start`, locate the anchor's index, and emit the genes at index offsets `−window … +window` (excluding the anchor itself). `max_bp_distance`, when set, further drops rows whose `bp_gap` exceeds it. (Validated query prototype in the design conversation.)

### Strand handling (strand is ~50% null)

- Every row always carries `strand` and a computed `same_strand` (`True` / `False` / `None`) — orientation is visible even when unfiltered.
- `same_strand=True` keeps only neighbors whose `strand == anchor.strand`; `same_strand=False` keeps only neighbors whose `strand` is the opposite. Either way, neighbors with null `strand` are dropped (orientation unconfirmable); the dropped count surfaces in the per-anchor envelope block (`dropped_null_strand`).
- If the **anchor's own `strand` is null**, the filter (`True` or `False`) is unappliable → emit a `warnings` entry and return the neighborhood **unfiltered** (rather than silently returning nothing).

### Per-row fields

`anchor_locus_tag`, `neighbor_locus_tag`, `rank_offset` (signed int; negative = upstream by `start`), `bp_gap` (unsigned intergenic distance anchor↔neighbor; `0` if the intervals overlap), `strand`, `same_strand` (`True`/`False`/`None`), `product`, `gene_name`, `gene_category`.

Decisions: **anchor excluded** from `results` (its metadata lives in the envelope block, not as a `rank_offset=0` row); `bp_gap` **unsigned** (direction is already carried by `rank_offset`'s sign); `window` default **±5**.

### Envelope

- Per-anchor block (`anchors`): `{locus_tag, organism_name, contig, start, end, strand, product, neighbors_returned, dropped_null_strand}`.
- `by_organism` — `{organism_name: neighbor_row_count}` rollup.
- `warnings` — e.g. anchor-strand-null-with-`same_strand=True`.
- `not_found` — `list[str]` anchor locus_tags absent from the KG.
- `not_matched` — `list[str]` anchors that resolve to a gene but lack coordinates (`start`/`contig` null) → no neighborhood computable.

### Edge cases

- Anchor near a contig end → fewer rows on the short side (no padding, no wraparound).
- Anchor alone on its contig → empty neighbor list, anchor still reported in the envelope block.
- Anchor lacks coordinates → `not_matched`.

## Cross-cutting conventions (per `docs://guide/conventions`)

- **`not_found` vs `not_matched`**: `not_found` = identifier absent from the KG; `not_matched` = identifier present but no qualifying data after the tool's intrinsic requirement (sequence / coordinates). Both are flat `list[str]` (single batch input each).
- Cross-organism batch is the default for both (locus_tags globally unique). No single-organism enforcement.
- Nullable fields (`strand`, `same_strand`) returned as explicit `null`, never elided or defaulted.

## Testing notes

Per the `testing` skill, all three layers covered:
- **Query-builder tests**: parameterized Cypher shape; `gene_neighbors` window/offset arithmetic, contig+organism scoping, `same_strand` predicate, `max_bp_distance` cutoff.
- **API-function tests**: envelope assembly (`sequence_length_stats`, per-anchor blocks), `not_found`/`not_matched` partitioning, `fasta` mode de-duplication (raw `sequence` absent when `fasta=True`).
- **Tool-wrapper tests**: Pydantic model validation, defaults (`window=5`, `fasta=False`), registration in `EXPECTED_TOOLS` / `TOOL_BUILDERS`.
- Use a fixture gene with null `strand` and a fixture gene with null coordinates to exercise the sparse paths.

## Decisions log

| Question | Decision |
|---|---|
| Sequence kind | Amino-acid (only sequence stored); tool named `gene_aa_sequence` (not `get_aa_fasta` — no `get_` prefix exists; `gene_` is the convention). |
| FASTA shape | `fasta` flag, default `False`. Sequence carried by rows **or** by the multi-FASTA blob, never both. |
| Neighbor window | Rank-based ±N (default `±5`) + optional `max_bp_distance` cap. |
| Same-strand filter | Optional `same_strand`; null-strand neighbors dropped when `True`; warn + unfiltered when anchor strand null. |
| Anchor in results | Excluded from `results`; reported in envelope per-anchor block. |
| `bp_gap` sign | Unsigned; direction via `rank_offset`. |
| Batch scope | Cross-organism for both (locus_tags globally unique). |
| KG changes | None — all fields already exist. |
