**Status:** design, approved to write plan
**Date:** 2026-05-01
**Scope:** slice 1 of N (3 new MCP tools + 2 existing-tool extensions; future slices for symmetric reaction-anchored tooling, dedicated pathway-chemistry surface, and the metabolite-DM pivot — deferred)

## Context

The knowledge graph chemistry layer (Phase 1.2 + 1.2.1 + 1.2.2) shipped on 2026-04-30 — full KG details:
`multiomics_biocypher_kg/docs/kg-changes/metabolism-chemistry-layer.md`.

Live state verified 2026-05-01:

- `Reaction` nodes: 2,349 (1,922 balanced / 427 unbalanced; all `reaction_class = "chemical"` today)
- `Metabolite` nodes: 2,188 (100% with `kegg.compound:C*` primary ID; 100% `mnxm_id`; 86% `chebi_id`; 54% `hmdb_id`; 31 lack `formula`)
- `Gene_catalyzes_reaction` edges: 52,742 across 21,117 genes; all 2,349 reactions are gene-reachable
- `Reaction_has_metabolite` edges: ~10,050 — **direction-agnostic** (substrates and products not distinguished)
- `Reaction_in_kegg_pathway` edges: 6,349 to 155 distinct pathways
- `Metabolite_in_pathway` edges: 8,095 to 310 distinct pathways
- `Organism_has_metabolite` edges: ~37,010 — post-import 2-hop materialization
- `OrganismTaxon.reaction_count` / `metabolite_count` rollups populated

The chemistry layer is currently queryable only via `run_cypher`. No first-class MCP surface exists.

## Driving research questions

From the user (marine-microbiology workflow):

1. **N sources MED4 uses** — given RNASEQ + proteome data, identify which nitrogen-containing metabolites Prochlorococcus MED4 metabolizes, with transcriptional response evidence under N-limitation.
2. **Cross-feeding Prochlorococcus / Alteromonas** — identify metabolites where one organism's catalysts go up in coculture and the other's also responds, suggesting metabolic exchange.

### Honest framing

The chemistry layer alone gets you to **capability**, not **active use**:

| Layer | Available today | What it answers |
|---|---|---|
| Chemistry (Phase 1.2, this spec's foundation) | ✓ live | Which genes catalyze which reactions; which metabolites do those reactions touch; which organisms carry the catalytic capacity |
| Transport (TCDB-CAZy spec, pre-development) | proposed, see [docs](../../../../multiomics_biocypher_kg/docs/kg-changes/tcdb-cazy-ontologies.md) | Which genes transport which metabolites — load-bearing for *uptake* questions like N-source biology |
| Metabolomics-DM (no spec yet) | future | Which metabolites are measured in which compartments / experiments — load-bearing for *direct evidence* of cross-feeding |

The driving questions are best answered by combining all three. Slice 1 ships the chemistry layer's MCP surface and is **forward-compatible** with the other two — same row classes and filter slots will populate transport rows on TCDB landing, with no explorer code change. Pure-metabolomics metabolites will appear in `list_metabolites` with `gene_count = 0` once that spec lands.

### KG-1: `Gene.reaction_count: int` post-import rollup

Single-hop count via `Gene_catalyzes_reaction`. Mirrors `Gene.expression_edge_count`. Slot in Group 3 of `scripts/post-import.sh`:

```cypher
MATCH (g:Gene)
CALL {
  WITH g
  OPTIONAL MATCH (g)-[r:Gene_catalyzes_reaction]->()
  WITH g, count(r) AS rxn_count
  SET g.reaction_count = rxn_count
} IN TRANSACTIONS OF 1000 ROWS;
```

### KG-2: `Gene.metabolite_count: int` post-import rollup — **defined as UNION across all gene-reaching paths**

Mirrors `Gene.compartments_observed` (existing 2-hop pattern). Initial implementation traverses catalysis path only; on TCDB-CAZy landing, the same property auto-extends to UNION the transport path with no explorer change.

```cypher
// Slice-1 (catalysis only):
MATCH (g:Gene)
CALL {
  WITH g
  OPTIONAL MATCH (g)-[:Gene_catalyzes_reaction]->(:Reaction)
                 -[:Reaction_has_metabolite]->(m:Metabolite)
  WITH g, count(DISTINCT m) AS met_count
  SET g.metabolite_count = met_count
} IN TRANSACTIONS OF 1000 ROWS;

// On TCDB landing, replace OPTIONAL MATCH with UNION of:
//   (g)-[:Gene_catalyzes_reaction]->(:Reaction)-[:Reaction_has_metabolite]->(m:Metabolite)
//   (g)-[:Gene_has_tcdb_family]->(:TcdbFamily)-[:Tcdb_family_is_a_tcdb_family*0..]->(:TcdbFamily {level_kind: 'tc_specificity'})-[:Tcdb_family_transports_metabolite]->(m:Metabolite)
```

The semantic definition stated up-front avoids a future breaking-change rename.

### KG-3: `Metabolite.elements: list[str]`

Hill-parsed presence list of element symbols (e.g. `H2O` → `["H", "O"]`; `NaCl` → `["Cl", "Na"]` with no false `["C", "H", "N"]`). Build-time, in `metabolism_adapter.py` or wherever Metabolite props are emitted. Empty list for the 31 metabolites without `formula`.

Eliminates the formula-regex footgun for element-presence queries. Filter becomes trivial Cypher: `WHERE 'N' IN m.elements`.

### KG-4: `KeggTerm.reaction_count` / `metabolite_count: int` post-import rollups

Pathway-level only (filter to nodes where `level_kind` indicates pathway). Single-hop count via `Reaction_in_kegg_pathway` and `Metabolite_in_pathway`. Slot next to existing chemistry rollups in `scripts/post-import.sh`:

```cypher
// Pathway reaction_count
MATCH (p:KeggTerm)
WHERE <pathway-level filter>
CALL {
  WITH p
  OPTIONAL MATCH (r:Reaction)-[:Reaction_in_kegg_pathway]->(p)
  SET p.reaction_count = count(r)
} IN TRANSACTIONS OF 100 ROWS;

// Pathway metabolite_count
MATCH (p:KeggTerm)
WHERE <pathway-level filter>
CALL {
  WITH p
  OPTIONAL MATCH (m:Metabolite)-[:Metabolite_in_pathway]->(p)
  SET p.metabolite_count = count(m)
} IN TRANSACTIONS OF 100 ROWS;
```

Slice-1 explorer surface does **not** consume these rollups directly — they're captured in the backlog (Tier-2 follow-up: surface in `list_metabolites.by_pathway` envelope rows + optionally on `genes_by_ontology(ontology="kegg")` pathway rows). The KG ask is requested now to avoid an extra rebuild later.

Distinct metabolites this family transports including subtree (so `tc_class` nodes show total substrate breadth across descendants). Lets `gene_ontology_terms(ontology="tcdb")` and `search_ontology` show substrate breadth at-a-glance — same UX as the existing `gene_count` rollup on every other ontology node.

**S5. (Optional) `TcdbFamily.tc_class_id: str | None` sparse pointer.**
BRITE precedent (`tree`/`tree_code`). Cheap class-level filtering without traversing the parent edge. Defer to a follow-up if not naturally in scope.

**Coordination note:** the removed `Gene.transporter_classification` array property may currently surface in `gene_details` (uses `g{.*}` pattern). Verify and update that tool as part of the TCDB landing — orthogonal to chemistry slice 1.

### For the future metabolomics-DM spec (no spec yet)

Pre-spec suggestions; will firm up when the metabolomics spec gets written.

**M1. `Metabolite.measurement_count: int` post-import rollup.**
Count of DM annotations on this metabolite. Mirrors `Reaction.gene_count` / `Gene.expression_edge_count`. Routing signal for `list_metabolites`.

**M2. `Metabolite.compartments_observed: list[str]` post-import rollup.**

**M3. `Metabolite.experiment_count: int` (optional).**
Distinct experiments where measured.

Formalize when metabolomics spec lands. Coordinate with TCDB-CAZy spec to use a stable enum across all three values.

**M5. Edge structure for measurements.**

## Tool specs

All three new tools follow the project field-design rubric being promoted in `add-or-update-tool` skill (Pass A): real-value examples in Pydantic `Field(description=...)`, drill-down clauses on routing fields, single-organism enforcement on detail tools, presence-only signposting where applicable.

### Per-node KG property categorization

Decisions for which properties surface where, based on the `OntologyTermRow` BRITE-pattern precedent (gene side = minimal; entity side = rich; organism_name verbose-only when single-organism enforced).

#### `Reaction` properties

| Property | Category | Reasoning |
|---|---|---|
| `id` (full, `kegg.reaction:R*`) | **result** as `reaction_id` | primary identifier |
| `name` | **result** as `reaction_name` | raw KEGG; field description documents the compound `<systematic_name>; <equation>` format. Equation order is **arbitrary in KEGG** — does not preserve substrate/product direction. |
| `ec_numbers` (list) | **result** | high-leverage routing — connects to existing EC ontology tools |
| `mass_balance` (`balanced`/`unbalanced`) | **result** | useful flag |
| `gene_count` (rollup) | **result** in `by_reaction` envelope; **skip** in row | row-level redundant |
| `mnxr_id`, `rhea_ids` | **verbose** | cross-refs, niche |
| `kegg_reaction_id`, `kegg_pathway_ids`, `reaction_class`, `organism_count`, `organisms`, `preferred_id` | **skip** | redundant, single-value-today, or KG convention |

#### `Metabolite` properties

| Property | Category | Reasoning |
|---|---|---|
| `id` (full, `kegg.compound:C*`) | **result** as `metabolite_id` | primary identifier |
| `name` | **result** | identity |
| `formula` | **result** | central chemistry signal |
| `mass` | **result** | useful narrowing/sorting |
| `elements` (KG ask 3) | **result** | structurally meaningful presence list; parallel to `Reaction.ec_numbers` |
| `chebi_id` (~86%) | **result, sparse** | high-leverage cross-ref to ChEBI ontology |
| `evidence_sources` (TCDB-CAZy spec adds) | **result** | path provenance: `metabolism` / `transport` / `metabolomics` (3 values when all paths populated) |
| `gene_count`, `organism_count` (rollups) | **result** in `MetaboliteResult` rows; **skip** in triplet rows | informative at the metabolite-discovery level |
| `transporter_count` (TCDB-CAZy spec adds) | **result** | informative when transport path populated |
| `kegg_compound_id` (raw) | **skip** | redundant with `id` |
| `inchikey`, `smiles`, `mnxm_id`, `hmdb_id` | **verbose** | structural / cross-ref fingerprints |
| `preferred_id` | **skip** | redundant |

#### `Gene` properties (when surfaced inside chemistry triplet rows)

Slightly looser than the strict BRITE pattern — adds two sparse-result fields for at-a-glance readability:

| Property | Category | Reasoning |
|---|---|---|
| `locus_tag` | **result** | identity |
| `gene_name` | **result, sparse** | often null but high-signal when present |

| `gene_category` | **verbose** | optional context |
| Bulky text (`function_description`, `gene_summary`, `all_identifiers`) | **skip** | LLM pivots to `gene_overview` for richer context |
| All rollups (`expression_edge_count`, DM counts, etc.) | **skip in chemistry rows** | `gene_overview` carries these |

#### `OrganismTaxon` chemistry properties

| Property | Category |
|---|---|
| `reaction_count` (rollup, KG already populated) | **result** |
| `metabolite_count` (rollup, KG already populated) | **result** |

`Gene_catalyzes_reaction`, `Reaction_has_metabolite`, `Reaction_in_kegg_pathway`, `Metabolite_in_pathway`, `Organism_has_metabolite` carry only an internal `id: str` (or empty). **Skip all.** Direction-agnostic disclosure for `Reaction_has_metabolite` is a docstring concern, not a field.

### Tool 1: `list_metabolites`

Discovery + filtering. Cross-organism by default. Filterable to one or more organisms via `Organism_has_metabolite`.

**Inputs**

### Cross-cutting decisions

**Direction-agnostic disclosure** (the central LLM-facing caveat).

Every chemistry tool docstring opens with:

> **Direction-agnostic.** This tool joins genes and metabolites through reactions (catalysis path) and, on TCDB-CAZy landing, through TcdbFamily (transport path). Neither path distinguishes substrates from products — neither at the edges (`Reaction_has_metabolite`, `Tcdb_family_transports_metabolite`) nor in the KEGG equation text (KEGG equation order is arbitrary). A gene that *produces* a metabolite and a gene that *consumes* it both surface in the same row. To distinguish, layer transcriptional evidence (`differential_expression_by_gene` direction) and gene functional annotation (Pfam / KEGG KO names like `*-synthase` vs `*-permease`). When the metabolomics-DM layer lands, it will provide measured-evidence direction.

**Naming consistency:** `metabolite_id` / `reaction_id` always carry the full prefixed ID (`kegg.compound:C00031`); `kegg_compound_id` / `kegg_reaction_id` carry the raw KEGG number (skipped from result rows; only in raw KG data).

**Triplet row class shared:** `GeneReactionMetaboliteTriplet` is used by both `genes_by_metabolite` and `gene_metabolic_role`. Identical row schema, different anchors.

**Forward-compatibility for the three Metabolite-evidence paths:**

| Path | Edge | Slice-1 row population | Notes |
|---|---|---|---|
| Catalysis | `Gene → Reaction → Metabolite` | populates today | the chemistry-layer foundation |
| Transport | `Gene → TcdbFamily → Metabolite` | populates on TCDB-CAZy landing | row class unchanged; `evidence_source = "transport"` per row |
| Metabolomics | `DerivedMetric → Metabolite` (likely) | does **not** produce triplet rows (no gene anchor) | pure-metabolomics metabolites appear in `list_metabolites` with `gene_count = 0` |

## Workflows

### Workflow A — N sources MED4 uses

Goal: identify nitrogen-containing metabolites whose catalyzing genes show transcriptional response under N-limitation in MED4.

```
1. Identify candidate genes (existing tools)
   differential_expression_by_gene(
     organism="Prochlorococcus MED4",
     experiment_ids=[N-limitation experiment IDs],
     direction="up", significant_only=True
   )
   → DE gene set (~50-200 locus_tags)

2. Filter to genes touching N-bearing metabolites (slice-1 chemistry)
   gene_metabolic_role(
     locus_tags=DE_genes,
     organism="Prochlorococcus MED4",
     metabolite_elements=["N"]
   )
   → triplet rows (gene × reaction × N-metabolite),
   summary `by_metabolite` ranks N-metabolites by gene breadth.

3. Sanity check (chemistry-anchored)
   list_metabolites(
     organism_names=["Prochlorococcus MED4"],
     elements=["N"],
     pathway_ids=["kegg.pathway:ko00910"]   # nitrogen metabolism
   )
   → catalogue of N-bearing metabolites MED4 is capable of.
```

**Honest framing:** the *uptake* side of N-source biology lives in the transport path — ammonium / nitrate / urea / amino-acid transporters are TCDB members, not catalysts. Slice 1 chemistry tools answer *internal N metabolism* (GS/GOGAT, AA biosynthesis) thoroughly and contribute partial answer to *uptake*. Full answer requires TCDB-CAZy spec landing. When it does, step 2 above will additionally surface transport rows with `evidence_source = "transport"` — no explorer code change needed.

### Workflow B — Cross-feeding capability between Prochlorococcus and Alteromonas

Goal: identify metabolites where one organism's catalyzing genes go up in coculture and the other's also responds, suggesting metabolic exchange.

```
1. Shared-capability metabolites (chemistry layer)
   list_metabolites(organism_names=["Prochlorococcus MED4", "Alteromonas macleodii ..."])
   → metabolites with organism_count >= 2 AND both organisms in the pair-of-interest
   (LLM filters per-row organism membership; or two single-org calls + intersect by metabolite_id)

2. For each shared metabolite, find catalysts on each side
   genes_by_metabolite(metabolite_ids=[shared_ids], organism="Prochlorococcus MED4")
   genes_by_metabolite(metabolite_ids=[shared_ids], organism="Alteromonas ...")
   → two gene sets per metabolite.

3. Score under coculture DE (existing tools)
   differential_expression_by_gene(
     organism=<each side>,
     locus_tags=<gene set from step 2>,
     experiment_ids=[coculture experiment IDs],
     direction="any"
   )
   → which catalysts respond, on which side.

4. Optional functional-annotation heuristic
   gene_overview(locus_tags=responding_genes_each_side)
   → per-gene product / Pfam to guess synthase vs transporter vs degradative.
```

**Honest framing:** "produces vs consumes" is unanswerable from chemistry alone. Slice 1 + DE answers *capability + transcriptional response*. Phase-2 metabolomics-DM (measured concentration changes in coculture supernatant) is the authoritative path for direction.

## Edge cases

### Empty-result distinctions (Pass A rubric clause #6)

| Scenario | Bucket | Where |
|---|---|---|
| Input metabolite ID malformed / not in KG | `not_found.metabolite_ids` | `genes_by_metabolite`, `list_metabolites` |
| Input metabolite ID exists in KG but has no catalyzing gene in this organism | `not_matched` | `genes_by_metabolite` |
| Input organism unknown | `not_found.organism_names` (or `not_found.organism`) | all chemistry tools accepting organism filter |
| Input locus_tag not in KG | `not_found.locus_tags` | `gene_metabolic_role`, `gene_overview` |
| Input locus_tag exists but no `Gene_catalyzes_reaction` edge | `not_matched` | `gene_metabolic_role` |
| Element symbol not used in any KG metabolite | `not_found.elements` | `list_metabolites`, `gene_metabolic_role` |

### Data sparsity

- **Reactions with empty `name`** (32/2,349). `reaction_name=None`. Field is nullable on triplet rows. Both `genes_by_metabolite` and `gene_metabolic_role` envelopes carry `reactions_without_name: int` when non-zero. Edges still join — these reactions are real.
- **Metabolites without `formula`** (31/2,188). `formula=None`, `elements=[]`, `mass=None`. Element filter behaves as "no match."
- **Genes without `gene_name`** — common; sparse-result design handles gracefully.
- **`Reaction_class != "chemical"`** (currently 0). Reserved for future transport reactions in the KG; chemistry tools today do not filter on it.

### Pathway scope asymmetry

`Metabolite_in_pathway` covers 310 pathways; `Reaction_in_kegg_pathway` covers only 155. A metabolite-anchored pathway query may return pathways that have no reaction-side edges (gene-reachable through compound annotation only). Tool docstrings note the asymmetry; LLM picks the join direction matching the question.

### Cross-feeding partner filter

`list_metabolites(organism_names=[A, B])` returns metabolites where AT LEAST ONE of A or B has the metabolite (UNION). The LLM either filters per-row by `organism_count` and inspects organisms, or runs two single-org calls + intersects by `metabolite_id`. Documented in the YAML `mistakes` section. An `intersection-only` mode is **not** baked into slice 1 (deferred to slice 2 if real workflow demands).

### Pure-metabolomics metabolites (forward-compatibility)

Once the metabolomics-DM spec lands, metabolites can exist in the KG with `gene_count = 0` (measured but no catalysis or transport). `list_metabolites` returns them; `genes_by_metabolite` and `gene_metabolic_role` return zero rows for them. Tool descriptions explicitly state: "*`gene_count = 0` does not mean the metabolite is absent from the KG — it may be present via measurement only.*"

## Open questions / risks

- **Direction-agnostic disclosure prominence.** This is the most important LLM-facing prose in slice 1. If it lands too quietly, the LLM may fabricate substrate/product distinctions. Mitigation: place the caveat at the *top* of every chemistry tool docstring (not at the bottom) and add a `mistakes` entry in each tool's YAML.
- **Triplet-row size.** With sparse fields (`gene_name`, `product`, `metabolite_chebi_id`), each row carries ~12 fields in result mode. For high-cardinality joins (a gene catalyzing 30 reactions × 4 metabolites each = 120 rows), responses can be large. Default `limit=25` controls; `summary=True` returns no rows; `verbose=True` adds another ~10 fields per row. Mitigation: document expected row volume and recommend `summary=True` for breadth questions.
- **`Gene.metabolite_count` UNION definition coordination.** If the TCDB-CAZy spec lands first and defines this differently (e.g. catalysis-only), there's a brief window of semantic divergence. Mitigation: pre-coordinate via this design doc's S3 schema suggestion before either spec ships.
- **Empty-name reactions.** 32 reactions have `name = ""`. They still produce edges and surface in queries. If a user runs an exact-text Lucene search, they're invisible. Acceptable today; revisit if KG team enriches downstream.
- **Cross-organism `list_metabolites` regression sensitivity.** Baselines are sensitive to organism-name ordering and KG-rebuild ties. Document the regen workflow explicitly in the regression test header.

## Note: `Gene.catalytic_activities` (out of slice-1 scope, future opportunity)

The 2026-05-02 KG rebuild that landed the chemistry-slice-1 asks also surfaced an existing-but-newly-noticed Gene property: `Gene.catalytic_activities: list[str]`. Populated on ~6,170 genes (max 13 entries per gene). Each entry is a UniProt-derived structured text record carrying:

- Reaction equation (text, with stoichiometry)
- **Rhea cross-reference** — Rhea IS direction-aware in the source data
- ChEBI cross-references for substrate/product metabolites
- EC number
- Evidence tag

Example (PMM0004 / purF):
```
Reaction=5-phospho-beta-D-ribosylamine + L-glutamate + diphosphate
       = 5-phospho-alpha-D-ribose 1-diphosphate + L-glutamine + H2O;
Xref=Rhea:RHEA:14905, ChEBI:CHEBI:15377, ChEBI:CHEBI:29985, ...;
EC=2.4.2.14; Evidence=
```

**Slice-1 treatment:** explicitly **not consumed** by chemistry slice-1 tools. The chemistry layer's graph-modeled `Gene_catalyzes_reaction → Reaction → Reaction_has_metabolite → Metabolite` traversal is the source of truth for gene-anchored chemistry. `catalytic_activities` is text, requires per-row parsing, and overlaps in coverage with the graph layer for catalytic genes that also have KEGG annotation. Surfacing it via `gene_overview` would inflate row size for limited routing benefit (`reaction_count > 0` already routes to `gene_metabolic_role`).

**Future opportunity (backlog):** parse `catalytic_activities` into structured edges with Rhea-derived direction information. This is the first place in the KG where substrate-vs-product direction is recoverable for catalytic activity. Pairs naturally with the future metabolomics-DM layer for full-direction-aware chemistry exploration. Out of scope for slice 1; tracked as a pre-spec opportunity.
